# 0011 — Work-tracking records are self-identified by UUIDv7; the backlog is a directory, not a file

Concurrent authoring of issues/PRDs/ADRs on separate branches collides in git two ways: `backlog.json` is a single array so every append fights the same closing bracket, and ids are locally-allocated sequential counters (`EXEC-NNN`, `PRD-NNN`) so two offline authors both grab the next number. We make records **self-identifying**: every new record carries a **UUIDv7** id, minted offline with no coordinator, and `backlog.json` is replaced by **one file per issue** under `.excn/issues/`, named `<uuid>-<slug>.json`. Two branches then never touch the same path and never contend for the same id, so the work-tracking set merges cleanly with zero infrastructure. UUIDv7's millisecond-timestamp prefix keeps records time-sortable; humans and agents reference records by their 8-char short prefix, the way they already reference commit SHAs.

## Considered Options

- **Keep sequential ids + stand up a coordinator** (server or merge-bot to hand out the next number) — the only way to preserve tidy `EXEC-091..096` ranges, but it forces standing infrastructure first and contradicts the framework's no-assumed-network posture. UUIDv7 removes the *reason* a coordinator existed.
- **Short random token behind the `EXEC-` prefix** (`EXEC-k7m2q`) — collision-free and grep-able, but keeps a bespoke scheme where a standard (RFC 9562) already fits.
- **Rewrite all historical ids to UUIDv7** — one clean format, but a bulk mutation of closed records to fix a problem they can never have again, and it would force `migrate` to touch work-tracking content (against ADR-0008). Rejected in favour of grandfathering.

## Consequences

- **Amends ADR-0005.** "`backlog.json` is the single issue tracker for an Instance" no longer holds — the `.excn/issues/` **directory is the tracker**. There is deliberately **no manifest/index file**; a list-of-all-issues file would reintroduce the exact monolith conflict. Readers and the viewer glob `.excn/issues/*.json`.
- **Supersedes the scribe-assigns-ids role in ADR-0007.** Ids are self-minted by the shared write helper, not assigned by `scribe` or the (out-of-scope) viewer-write server. `scribe` remains the record *owner*; it is no longer the id *authority*.
- **Grandfathering.** Existing `EXEC-NNN` / `PRD-NNN` / sequential ADR ids are kept as-is; schema id-patterns widen to accept legacy **or** UUIDv7. New records always mint UUIDv7. This ADR is itself numbered sequentially (`0011`) under that rule — the cutover is forward-only.
- **The helper is the exclusive write path for issues and sprints.** A shared `writeRecord` helper accepts a JSON record, validates it against the schema with a **bundled** ajv, writes atomically, and emits canonical serialization. For issues it mints the UUIDv7 and **rejects a caller-supplied id** (the id is immutable; other fields stay updatable). For sprints it emits canonical-sentinel form (arrays-last, sentinel dead-last) so the trick that blunts concurrent `step_log` appends actually holds — one serializer, one byte-layout — and it exposes a `step_log` *append* mode so a gate adds one verdict without rewriting the file. **PRDs** (JSON, authored by `execution-to-prd`) and **ADRs** (markdown, authored in the grill) are created manually, each minting its own UUIDv7; they are low-churn, already one-file-per-record, and not the conflict hot path. Routing issues and sprints exclusively through the helper makes the malformed record unconstructible rather than merely discouraged — the lesson of the trailing-comma hand-edit defect.
- **Enforcement is one thin channel guard, not a content re-check.** Because the helper writes via the filesystem and never the agent's `Write` tool, any raw `Write`/`Edit` targeting `.excn/issues/` or `.excn/sprints/` is by definition a bypass. A single `PreToolUse` guard — modeled on `progress-location-guard`, a hard deny whose reason redirects to `to-execution …` — denies raw writes to those two homes and needs no knowledge of uuids, shapes, or schemas. It **subsumes** the monolith-shape ban and the uuid/serialization checks: format is guaranteed by construction at the helper, so the hook stops duplicating schema logic that must otherwise track the schema — no ajv, no shape predicate, no per-write validation (the surface and token reduction). PRDs/ADRs, authored manually, are not channel-guarded; **process-adherence** at the lifecycle transition is the set-level backstop (orphans, references, a manual sequential-id regression) for every artifact type.
- **Migration is a distinct one-time command**, not `migrate` (which stays location-only per ADR-0008): it splits `backlog.json` into per-issue files and retrofits existing sprint files to canonical-sentinel form, verifying every write re-reads schema-valid **before** deleting the monolith, idempotent and absolute-path-resolved (the phantom-wipe lesson). All inputs are committed, so a botched run is `git checkout`-recoverable.
- **No coordinator/server is in scope.** UUIDv7 makes the write-capable viewer server (EXEC-056) a live-editing UX feature, fully decoupled from conflict-avoidance.
- **What is given up:** the tidy sequential integer and "EXEC-091..096" range shorthand. Chronological order survives via the UUIDv7 timestamp prefix; contiguous numeric ranges do not.
